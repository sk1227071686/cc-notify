# CC Notify — Retrospective & Lessons Learned

## 遇到的问题及避免方法

### 问题 1：Nginx conf.d 不被主配置加载

**现象：** 在 `/etc/nginx/conf.d/wecom-proxy.conf` 中创建了代理配置，`netstat -anp | grep 8443` 未显示监听端口。

**根因：** 主配置 `nginx.conf` 中没有 `include /etc/nginx/conf.d/*.conf;`，导致 conf.d 下的配置文件不会被加载。

**解决：** 将配置直接追加到 `nginx.conf` 的 `http {}` 块中。

**后续避免：**
- 修改 Nginx 配置后，必须验证端口是否实际监听（`netstat -anp | grep <port>`），不能只看 `nginx -t` 通过就以为生效
- 首次使用一台服务器时，先检查 `nginx.conf` 的 include 结构：`grep include /etc/nginx/nginx.conf`

---

### 问题 2：Nginx 代理 400 Bad Request — 代理指向错误的下游

**现象：** 配置 Nginx 8443 全部路径 `proxy_pass http://127.0.0.1:8444`（Python 回调服务），导致本机通过代理访问 `/cgi-bin/gettoken` 返回 400。

**根因：** Python 回调服务只能处理 `/callback/*` 路径的 GET 验证请求和 POST 消息请求，不识别 `/cgi-bin/*` 路径，返回 400 Bad Request。

**解决：** 在 Nginx 中按路径分流：
- `/callback/*` → `http://127.0.0.1:8444`（Python 回调服务）
- 其他路径 → `https://qyapi.weixin.qq.com`（企微 API 代理）

**后续避免：**
- 配置反向代理时，必须根据下游服务的能力划分路径，不能将所有流量指向一个非通用服务
- 标准模式：反向代理多个 location，每个指向各自的后端服务

---

### 问题 3：企微回调 URL 验证失败 — 直接返回原始 echostr

**现象：** 第一次尝试用 `return 200 "$arg_echostr"` 让 Nginx 直接返回 echostr 参数，企微验证不通过。

**根因：** 企微回调 URL 验证要求**解密 echostr 后返回明文**，而不是直接返回加密字符串。根据[官方文档](https://developer.work.weixin.qq.com/document/10514)：
1. 企微发 GET 请求，带加密的 echostr 参数
2. 服务器用 AES-CBC 解密 echostr
3. 从解密结果中提取 msg 字段（去掉 16 字节 random + 4 字节 msg_len）
4. 1 秒内原样返回明文 msg 成熟的错误：没有先阅读官方文档，凭直觉实现验证逻辑

**解决：** 部署 Python 回调服务（`/opt/wecom-callback/server.py`），使用 `pycryptodome` 实现 AES 解密。

**后续避免：**
- **先阅读官方文档再动手**：任何涉及第三方 API 验证/签名/加解密的逻辑，必须先读完官方文档再写代码
- 对于验证类流程，先用官方调试工具测试，再集成到代码中

---

### 问题 4：企业可信 IP 必须先设置回调 URL

**现象：** 想直接配置可信 IP，但界面上提示「请先设置可信域名或设置接收消息服务器URL」。

**根因：** 企微的安全模型要求先验证回调 URL 可达性，才开放可信 IP 配置。这是一种安全设计——确保企业确实拥有该服务器。

**解决：** 部署回调验证服务，通过 URL 验证后再配置可信 IP。

**后续避免：**
- 涉及第三方平台的安全配置，提前了解完整的配置依赖链（A 需要 B 先完成，B 需要 C 先完成）
- 向用户引导时，先讲清完整的依赖关系再开始，避免做到一半卡住

---

### 问题 5：notify.sh 端口号写错

**现象：** notify.sh 中 `WEDEX_PROXY="https://goods.fatrabbits.shop:8444"`，访问企微 API 时返回 400。

**根因：** 8444 是 Python 回调服务的端口（只能处理 `/callback/*`），8443 才是 Nginx 对外代理端口。写脚本时用了错误的端口号。

**解决：** 修改为 `8443`。

**后续避免：**
- 脚本中的端口号应该从配置文件读取，不应硬编码
- 部署多服务时，在注释或文档中明确每个端口的用途

---

### 问题 6：access_token 获取失败时脚本 exit 1 阻塞 CC

**现象：** 原版脚本获取 token 失败时执行 `exit 1`，导致 CC hook 管道阻塞（没有输出 `{}`）。

**根因：** CC 的 hook 协议要求脚本必须输出 `{}` 到 stdout，无论成功与否。`exit 1` 在 `set -euo pipefail` 下会提前终止，跳过最后的 `echo "{}"`。

**解决：** 所有错误路径都先 `echo "{}"` 再 `exit 0`。原则：**永不阻塞 CC 管道**。

**后续避免：**
- CC hook 脚本必须在所有退出路径上都输出 `{}`
- **不使用 `set -e`**（或改为 `set -uo pipefail` 不带 `-e`），避免意外提前退出
- 错误信息只写 stderr，stdout 只写 `{}`

---

## 经验总结：开发规范建议

以下是从本次开发中提炼的规范，适用于所有 CC hook/skill 开发：

### 规范 1：Hook 脚本安全原则

1. **永不阻塞**：所有退出路径必须输出 `{}`
2. **不使用 `set -e`**：用显式错误处理代替
3. **错误只写 stderr**：stdout 只给 CC 协议需要的 `{}`
4. **exit 0 永远**：hook 失败不应让 CC 认为出错

### 规范 2：第三方 API 集成原则

1. **先读文档再动手**：特别是验证/签名/加解密流程
2. **摸清配置依赖链**：了解 A→B→C 的前置关系再引导用户
3. **用配置文件，不硬编码**：所有凭证从外部读取
4. **代理架构提前规划路径分流**：不同后端服务用不同 location

### 规范 3：部署引导原则

1. **告诉用户完整步骤再开始**：先展示依赖链全景，再逐步执行
2. **每步验证**：做完一步立即验证（`nginx -t`、`netstat`、`curl` 测试）
3. **错误信息要可操作**：不仅说"失败"，还说"可能原因和修复方法"
4. **原子写入**：配置文件用 temp+mv，不留半成品

### 规范 4：Skill 文档原则

1. **SKILL.md 要包含完整部署指南**：不要假设用户知道任何前置知识
2. **提供两种路径**：自动向导 + 手动配置，满足不同偏好
3. **消息格式要写明白**：给出实际的示例消息内容
4. **故障排查表**：问题→原因→修复 三列对照表
